# v1.38.4: Drop Embedded availableModels from chatroom_machines

## Overview

In v1.38.4, the `availableModels` field has been extracted from the `chatroom_machines` table into a new dedicated `chatroom_machineModels` table. The field was ~50 KB per machine and lived on the parent row, causing it to be re-pushed to every `listMachines` subscriber on any machine-row write (heartbeat, status change, etc.). Separating it eliminates the unnecessary payload from the high-frequency `listMachines` subscription.

## What Changed

| Area | Change |
|------|--------|
| `register` + `refreshCapabilities` mutations | Dual-write: keep legacy `chatroom_machines.availableModels` write AND upsert into new `chatroom_machineModels` table (one row per machine, all harnesses). No-op write suppression prevents subscription churn when model list is unchanged. |
| `listMachines` query | `availableModels` removed from response shape — payload no longer rides on re-pushes. |
| `getMachineModels` query (new) | Reads from `chatroom_machineModels` first; falls back to legacy `chatroom_machines.availableModels` for machines not yet back-filled by migration. |
| Schema | `chatroom_machines.availableModels` kept as `@deprecated optional` for backwards compatibility with old daemons + read tolerance. Will be removed after migration. |
| Frontend | `AgentConfigTabs` migrated to `useMachineModels(machineId)` hook (thin wrapper over `getMachineModels`). No longer reads `MachineInfo.availableModels`. |

## Required Deploy Step

After deploying v1.38.4, run the migration to strip the embedded `availableModels` payload from existing `chatroom_machines` rows:

```bash
cd services/backend
npx convex run migrations:run '{"fn":"migrations:dropEmbeddedAvailableModels"}'
```

Or run all migrations in order:

```bash
cd services/backend
npx convex run migrations:runAll
```

The migration is **idempotent** — safe to re-run. Rows that have already been cleaned (`availableModels === undefined`) are skipped automatically.

### Rollout Safety

The migration is safe to run before all daemon clients have upgraded:
- New daemons (v1.38.4+) dual-write into both `chatroom_machines` and `chatroom_machineModels` on every `register` / `refreshCapabilities` call.
- Old daemons (pre-v1.38.4) only write `chatroom_machines.availableModels`. If the migration clears the legacy field before those daemons upgrade, `getMachineModels` will return `{}` until the daemon's next `refreshCapabilities` call — which re-populates `chatroom_machineModels` via the dual-write path.

## Schema Cleanup (Future)

Once `dropEmbeddedAvailableModels` has been run against every environment that contains `chatroom_machines` rows, the `availableModels` optional field can be removed from the `chatroom_machines` validator in `schema.ts` entirely. This is left for a future PR.
